I made a complicated game in 2 weeks. How did I do?

It went about as well as expected. I got the base game working by the skin of my teeth, which is a really weird phrase because now I'm picturing a tooth wearing a hoodie made of skin. Instead of getting totally distracted by this train of thought, let me summarize what the original concept for this game was. Some time ago I decided it would be cool to make a Virtual Reality thingymabob that I could chuck a load of random mechanics into as a way to get more familiar with Unreal Engine and also as a way to attempt to justify the impulse selling of my soul to Mark Zuck-on-deez-nutz-erberg. I decided to use my last coursework assignment this year as an excuse to work on this, as the assignment itself was pretty open ended. However, that meant that it needed to be somewhat coherent, so the original idea kind of fell apart and I picked up some of the pieces to create ChronoloGuard instead. As I mentioned previously: "Chronologuard is a VR-PC co-op game about clearing the void of the temporal shadows left by the time travelling fools who were unfortunate enough to paradox themselves into oblivion." The VR player is tasked with navigating the level, finding and crafting weapons, and collecting currency by slaying enemies. The PC player gets to see all this from a birds eye view. Their job is to help by altering the level layout, thereby controlling the flow of enemies.
Considering I only gave myself 2 weeks to do all this, I think I succeeded in my goal of "pulling a miracle out of my arse". The one constantly recurring issue was collision. I set up a collision channel for the building grid, but not for anything else. I guess I thought it wasn't necessary? It absolutely was necessary. Especially for a VR game where the interactions are more likely to be more physical thereby relying more on checking for hits and overlaps. My improperly configured collision affected the building tile updates, building placement, enemy pathfinding, VR teleporting, and even the crafting system. Such a headache!
Towards the end I was just shoving workarounds for collision and other stuff on top of the existing stuff. This was the second biggest issue. The classes became unweildy, unreadable, and super tangled up. I don't think I could remove or significantly change one part of the game without several other things mysteriously exploding. Should be using interfaces and proper referencing instead of the 500 casts and custom events I have now. The last main issue is that I probably should have given myself more than two weeks because that was unnecessarily stressful. I'll figure out this "time" thing someday.

Hello [object Object] my old friend...

I've started making the tag system and have actually already got it working! Here's how it works:
1) Each post is an article element with a data attribute called "tags" that contains all the tags for the post.
2) The search page has a javascript function that takes a file (this page) and a query (the tag you're searching).
3) When run, it gets the html from the file and gets every article that has a data attribute containing a tag that matches the query.
4) Finally, it loops through the collection of articles it found and adds them all to the page.

The only problem with this is that for it to work for every post it has to load every page all at once. So eventually loading is going to be pretty slow if I continue for long enough, which I might! I've been kinda enjoying this lil daily project.
